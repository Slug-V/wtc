# CLAUDE.md

このリポジトリで作業する AI アシスタント（および将来の自分）向けの実務ガイド。
コードだけからは読み取りにくい背景・原則・落とし穴をまとめる。

## プロジェクト概要

「8時間勤務してもよい6時間契約」の働き方で**月給を一定に保つ**ための勤務時間管理 PWA。
実体は単一の `index.html`（HTML / CSS / JS を全内包）のみ。ビルド工程・外部ライブラリ・
サーバーは無い。データは端末の `localStorage` に保存し、バックアップは JSON の手動
書き出し／読み込み。

## 最重要の原則（壊さない・触らない）

1. **単一ファイル厳守**。HTML/CSS/JS はすべて `index.html` 内に置く。外部ライブラリ・
   ビルド・サーバー・パッケージ管理を導入しない。
2. **金額・時刻をソースに残さない**。時給・月給・勤務時刻などの個人値をコードに
   ハードコードしない（公開リポジトリのため）。ユーザー入力 → `localStorage` のみ。
   新機能で個人値を扱う場合もこの原則を守る。
3. **モード表示を動的化しない**。6h残枠／8hノルマの表示モードは年度データの時点で各月
   固定。月の途中で「残枠が減ったらモードを切り替える」等は設計意図に反する。提案も不要。
4. **GitHub Pages のサブパス配信前提**。リソース参照に先頭スラッシュの絶対パスを使わない
   （`/foo` ではなく相対参照）。manifest の `start_url` も `"."`。
5. **localStorage スキーマは非破壊**。既存利用者が実データを記録中。キーのバージョン
   （`...v1`）を尊重し、構造を変えるならマイグレーションを必ず用意する。`settings` への
   キー追加のような後方互換な拡張を優先する。
6. **検証は iPhone 単体でできる粒度**で、一機能ずつ。後述の「検証」を参照。

## ドメイン仕様（コードから読み取りにくい背景）

- **基準**：6時間契約・基準日数 `BASE_DAYS = 18`（年1更新、今年度は18で確定）。
  基準18日を全て8hで埋めると月給（時給 × 8h × 18日）に一致する。**整合性の検算点**。
- **6h残枠モードと8hノルマモードは同一状態の別表示**。別の雇用状況ではない。月内の
  6h化日数が総勤務日数の半分以下なら6h残枠、半分超なら8hノルマで表示（小さい数を見せる）。
  判定は `YEAR_DATA` の時点で各月固定。
- **基準割れ月は残業ノルマモード（`overtime`）**。残業でしか月給を埋められない月。
  残業は割増1.25倍なので、要残業時間 = 不足時間 ÷ 1.25。ゲージで進捗表示。
  （年度内では8月が該当。「無理に平準化しない判断もあり」と本人合意済み。）
- **有休は6h相当で給与確定**する（フルタイム8h扱いではない）。よって有休も6hボタンで
  記録し、金額も6h分で正しい。区別不要が正しい。
- **残業の割増は1.25倍。欠勤は無給（0円）**。
- **欠勤の補正（独自ルール・要保持）**：
  - 6h残枠月の欠勤 … 残枠が**計4減る**（枠1消費 + 埋め合わせ3日分）。
  - 8hノルマ月の欠勤 … ノルマが**3増える**（上記の裏側の表現）。
  - 残業ノルマ月の欠勤 … 要残業に8h（480分）分を上乗せ。
  - 欠勤は実績上まだ発生していない（数学上の選択肢として実装）。

## アーキテクチャ / 主要関数（すべて `index.html` 内）

データ:
- `YEAR_DATA` … 年度別の月データ。`mode`（`6h-quota` / `8h-quota` / `overtime`）、
  `workdays`、`sixDays`、`eightDays`、overtime月のみ `otHours`。範囲は 2026-04〜2027-03。
- `records` … `{ "YYYY-MM-DD": "6h"|"8h"|"8h1h"|"8h2h"|"absent" | {k:"8hX",min:数値} }`。
  端数残業のみオブジェクト（`min` は分）、それ以外は文字列。キー `worktime.records.v1`。
- `settings` … `{ wage:数値, schedule:{start,breakStart,breakEnd} }`（いずれも任意）。
  キー `worktime.settings.v1`。`schedule` の時刻は `"HH:MM"` 文字列。

集計・描画:
- `recKind(v)` / `recLabel(v)` / `recTagClass(v)` … 記録値の正規化と表示。
- `computeMonth(mk)` … その月の記録を集計（six/eight/absent/otMin/earned）。**金額(earned)
  のみ「今日は勤務完走後に算入」のゲートが入る。日数カウントはタップ即時のまま。**
- `metricFor(mk)` … 残枠／ノルマ／残業ゲージの計算（カウントのみ使用、時間ゲートの影響なし）。
- `render()` … ヘッダ・今日の所定・メトリクス・確定額を再描画。末尾で `refreshTodayEarn()`。

勤務時間帯 & 今日の発生額（リアルタイム）:
- `parseHM`, `getSchedule` … 時刻文字列の解釈。休憩は両方揃い開始<終了のときのみ有効。
- `kindMinutes(v)` … 種別 → 支払対象の基本分/残業分/休憩有無（6h=360分・休憩なし、
  8h系=480分・休憩あり、残業分を加算）。absent/未記録は null（稼ぎなし）。
- `buildTimeline(v,sch,wage)` … 時計上の加算セグメント（分単位・円/分）を組む。休憩帯は
  隙間にして加算停止を表現。残業帯は1.25倍レート。
- `earnedAt(segs,nowMin)` … 現在時刻までの発生額を積分。
- `todayMoneyConfirmed()` … 今日の金額を今月確定額に算入してよいか。時間帯未設定なら
  従来どおり即時、設定済みなら勤務完走（終業時刻到達）後のみ true。
- `todayEarnState()` … 表示可否・発生額・状態（開始前/勤務中/休憩中/本日終了）・完走フラグ。
- `refreshTodayEarn()` / `startTicker` / `stopTicker` … ピルの更新と1秒ティック制御。
  表示中のみ稼働、非表示・完走後は停止。完走の瞬間に一度だけ `render()` を呼び確定額へ反映。

## 「今日の発生額」と「今月確定額」の関係（重要な不変条件）

- 勤務中：今日分の金額は**ピルにのみ**存在し、今月確定額には算入されない（ピルは金色・
  「確定前」）。
- 勤務完走後：今日分が今月確定額に**離散的に算入**され、ピルは緑・「✓ 確定・今月計に算入」
  に切り替わる（＝確定額に入った旨の再掲。二重計上ではない）。
- 残枠／ノルマの**日数カウントはタップ即時**で動く（計画用なので時間ゲートしない）。
- 整合性：`buildTimeline` の総額（終業到達時）は `computeMonth` の1日分支給額と一致する。
  これは変更時の検算点。

## データ構造とバックアップ

- バックアップ JSON は `records` と `settings` の両方を含み、import は両方復元（寛容に処理）。
- `schedule` は `settings` への追加キーであり、旧バックアップに無くても「未設定」として扱える
  （マイグレーション不要）。スキーマを変える場合は `...v1` のバージョンと移行を必ず考慮する。

## 検証

- **環境制約**：平日は iPhone のみ、フル機能 PC は週末等に限られる。よって「iPhone 単体で
  確認できる粒度」の変更が望ましく、一度の大改修で「PC でないと直せない壊れ方」を避ける。
  変更は一機能ずつ・明確な単位で。壊れたときに戻せるようコミット単位を意識する。
- **ブランチのプレビュー**：単一の静的 HTML なので、ブランチを githack 等で開けば実機 Safari
  で確認できる（raw は `text/plain` のため不可）。例：
  `https://raw.githack.com/<owner>/wtc/<branch>/index.html`。本番 URL とは別オリジンのため
  localStorage は分離され、本番データに影響しない。
- **ヘッドレス確認**：このリポジトリには Chromium + Playwright が利用できる環境がある
  （`PLAYWRIGHT_BROWSERS_PATH=/opt/pw-browsers`、`playwright install` は不要）。
  時刻依存の挙動は `page.clock.install({ time })` で固定し、`addInitScript` で `localStorage`
  を仕込んでから `file://` で開く。検算は「`buildTimeline` 総額 == `computeMonth` の支給額」。

## 既知の拡張候補

- 着地予測額（残り日を配分どおり消化した場合の月末見込み額）
- 通常月での残業ボタン（部分消化）対応
- 訂正シートでの端数（`8hX`）編集（現状は現値維持のみ）
- Git 同期 / クラウドバックアップ
- 来年度の最適基準日数を検討する別アプリ（JSON ビルダー）

## コミット / 作業の作法

- 単一ファイルゆえビルド・テストランナーは無い。動作確認は上記「検証」に拠る。
- **チャットや作業セッション固有のラベル・符牒をコード／コミットメッセージに残さない**
  （対話中に選択肢を指すための内部呼称など、その場限りの語）。コメントは、文脈を知らない
  読み手にも恒久的に意味が通る記述にする。
- 変更は一機能ずつ。コミットメッセージは日本語で、何を・なぜ変えたかを簡潔に。
